AI Coding Agent Engineering & UI/UX Specification
This document provides a highly structured engineering and UX blueprint for building an autonomous AI coding agent. It covers workspace file actions, secure sandboxing, OAuth-based integrations, task scheduling, and dynamic UI state representation via collapsible accordions.
1. Core Agent Workflows & Workspace Integrations
1.1 File Creation & Live Preview System
To prevent unstable mutations and ensure seamless frontend feedback, the agent and the runtime platform must collaborate through a strict creation-to-preview lifecycle.
code
Code
[Agent Path Resolution] ──> [Structured Tool Call] ──> [Compilation/Verification]
                                                               │
[Render UI Iframe]  <──  [Port-Forwarding Proxy] <──  [Dev Server Background Run]
File Creation Workflow
Path Resolution: Before performing a write operation, the agent queries the directory tree. It determines the correct framework directory structure to ensure files are saved in logical sub-directories (e.g., placing React components in src/components/ and not the root directory).
Structured Tool Invocation: The agent avoids executing raw shell commands (like echo "code" > file.js) as they are prone to escaping issues. Instead, it calls a dedicated system-level workspace API:
code
JavaScript
writeFile(path: string, content: string)
Verification: The agent immediately validates the write operation by executing a syntax check or reading the file back to confirm integrity.
Live Preview Initialization
Dependency & Command Analysis: The agent parses workspace configurations (such as package.json or cargo.toml) to locate the appropriate developer server startup command (e.g., npm run dev, vite, or python3 -m http.server).
Asynchronous Daemon Process: The platform executes the dev command as an asynchronous background process that survives across the agent's turn-taking cycles.
Dynamic Port Detection: The host system (the container engine or VM hypervisor) monitors network interfaces. When a local port starts listening (e.g., port 3000 or 5173), the host interceptor captures the port mapping.
Proxy URL Mapping & IFrame Rendering: The host maps the internal port to a public, secure proxy address. The chat client intercepts this event and opens a "Live Preview" panel containing an iframe pointed to that proxy address. The agent is notified that the preview is live, enabling it to update the user.
1.2 Secure Secrets & API Key Management
Typing API keys or secrets directly into a chat input is a security risk. The system must segregate credentials from the LLM prompt logs.
code
Code
[Agent Halts] ──> [UI Form Rendered] ──> [User Inputs Key] ──> [Encrypted Env Vault]
                                                                        │
[Agent Resumes] <──────────────────────────────────────── [Decrypted Key Read via env]
Contextual Halt: When the agent detects that a required integration key (such as OPENAI_API_KEY or DATABASE_URL) is missing, it halts execution immediately.
UI Payload Trigger: Instead of prompting for a text response in the chat, the agent outputs a structured UI command specifying the needed variable names. The frontend catches this command and renders a secure input form in the chat pane.
Vault Injection:
The user submits the secret through a masked password field.
The platform writes this value directly to the environment's secure vault (an encrypted key-value store or a .env file added to .gitignore).
Context Resumption: Once the secret is written, the platform triggers a signal that updates the runtime environment context. The agent resumes execution, reading the decrypted variable at runtime (e.g., process.env.API_KEY) without exposing it in the plain-text dialogue history.
1.3 GitHub Integration & Credential Handling
The agent should never ask for a user's plaintext password or personal access tokens directly. It must rely on system-mediated OAuth scoping.
Authentication Handshake: The host platform provides a native GitHub OAuth application. The user grants access, permitting scoped writes on designated repositories.
Token Retrieval & Injection: When the user initiates a repository task, the platform requests an installation access token or OAuth token.
Scoped Token Injection: The platform securely injects the token as a temporary environment variable (e.g., GITHUB_TOKEN) inside the agent’s execution sandbox.
Isolated Git Operations: The agent interacts with Git via CLI tools configured with authentication helpers, or by formatting remote URLs to leverage the environment variables directly:
code
Bash
git clone https://x-access-token:${GITHUB_TOKEN}@github.com/username/repository.git
This process keeps raw access tokens isolated from both user chats and LLM generation traces.
2. Execution Architecture & Shell Sandboxing
2.1 Sequential Task Execution Lifecycle
To guarantee robust automation, the agent divides user requests into linear stages and runs them within a strict self-healing loop.
code
Code
[ User Prompt ] ──> [ 1. Plan Phase ] ──> [ 2. Validate ] ──> [ 3. Step-by-Step Execution ]
                                                                        │
                                   [ Success ] <── [ Compile & Test ] <─┘
                                        │
                                   [ Next Step ]
1. Planning Phase: The agent analyzes the instruction set and breaks it down into sequential, high-level milestones.
2. Milestone Breakdown: Every milestone is divided into granular, actionable steps (e.g., installing a package, writing a handler, running test commands).
3. Linear Execution: The agent executes steps one-by-one. It must block the execution of subsequent steps until the current step yields a successful exit state.
4. Self-Healing Loop: If a step fails, execution halts. The agent is automatically fed the exit logs and compiler errors. It must generate a patch, apply it, and re-verify the step before the runtime allows the sequence to move forward.
2.2 Background Command Execution & Sandboxing
All terminal operations are isolated to protect host resources and capture logs cleanly.
Sandboxed Environment: The agent runs commands in an isolated environment, such as a secure Linux container or hypervisor-managed microVM.
Asynchronous Streams & Multiplexing: The agent passes commands to a background execution daemon. The daemon instantiates a pseudo-terminal (PTY) and streams both standard output (stdout) and standard error (stderr) to the client-side UI in real time.
Exit Code Evaluation: Every shell execution returns an integer exit code on completion:
Exit Code 0: Success. The agent proceeds to the next planned step.
Non-Zero Exit Code: Failure. The platform pauses execution, alerts the user, and hands the failure logs back to the agent for debugging.
Interruption Signals (SIGINT/SIGTERM): To prevent infinite loops or hanging servers from exhausting container resources, the UI and the agent must have the ability to send kill signals (such as SIGINT / Ctrl+C) to active background processes.
3. Front-End UI Behavior (Accordions & Collapsible Cards)
To maintain a clean chat canvas and manage dense build output, the frontend encapsulates log outputs using collapsible cards.
3.1 The Collapsible Card UX Pattern
Parent Group: A collapsible card that acts as a container for a logical set of steps (e.g., "Planning deployment completion verification", "Cloning Sovereign_Lab repository").
Meta Header: Displays status badges (such as a spinning icon for running, a green checkmark for success, or a red alert sign for failure) along with an action count (e.g., "3 actions" or "2 minutes ago").
Collapsible Content: The individual terminal logs and file differences are hidden inside the container.
Auto-Collapse & Auto-Scroll:
While Running: The card is automatically expanded and auto-scrolls down to display the incoming standard output.
Upon Success: The card collapses into a compact status pill to prevent clutter, preserving the main chat stream's readability.
3.2 State-Driven Accordion Behaviors
The visual representation of an accordion changes depending on the execution state of the tasks grouped within it:
Task State	Accordion Behavior	Visual Indicators
Pending	Collapsed and locked.	Grey text, clock icon, or empty circle.
Running	Auto-expands to display real-time stdout/stderr outputs.	Spinning loader, blue/purple highlight, pulsing active line indicator.
Success	Auto-collapses once execution concludes successfully.	Green checkmark, execution duration badge (e.g., Took 12s).
Failed	Remains expanded or auto-expands to draw attention to the error.	Red alert badge, error logs auto-scrolled into center view.
Nested Command Accordions
Inside each major task group (e.g., "Building Application"), individual command executions should use nested sub-accordions. Clicking a parent group opens the list of commands (e.g., npm run build), and clicking the command displays its raw execution terminal logs.
4. Agent Control Prompts (Instruction Sets)
Use one of the two prompts below to direct your coding agent on how to output structured data compatible with collapsible accordion renderers.
Approach A: Custom XML Markdown Prompt (For Markdown Chat Renderers)
This approach is ideal if your chat interface parses custom HTML/XML tags within the agent's markdown response. [1]
code
Xml
<system_instruction>
You are an advanced software engineering agent. When executing tasks, running commands, or auditing files, you must structure your chat outputs using custom collapsible markdown tags so the UI can render them as accordions.

Do NOT dump raw command outputs directly into the chat. Always wrap them in the following XML/HTML structure.

### Syntax Rules:
1. Every major milestone must be wrapped in a `<task-group>` element.
2. The `<task-group>` must have a `title` and a `status` ("running", "success", "failed", or "pending").
3. Individual terminal commands and their outputs inside the group must be wrapped in a `<command-log>` element.

### HTML/XML Structure Template:
<task-group title="Installing Dependencies" status="success" actions="2">
  <summary>Successfully verified package.json and installed node_modules.</summary>
  
  <command-log command="npm install --frozen-lockfile">
    added 142 packages, and audited 143 packages in 4s
    found 0 vulnerabilities
  </command-log>
</task-group>

<task-group title="Running Build Pipeline" status="failed" actions="1">
  <summary>Build failed during compilation due to a syntax error in main.ts.</summary>
  
  <command-log command="npm run build">
    > vite-project@0.0.0 build
    > tsc && vite build
    
    src/main.ts:12:15 - error TS1005: ';' expected.
  </command-log>
</task-group>

Apply this layout to all code modifications, repository audits, tests, and deployments you perform.
</system_instruction>

## 5. Client-Side Implementation Blueprint

Below is an implementation template demonstrating how a React application parses and renders the state-driven `TaskGroup` accordion.

```typescript
import React, { useState, useEffect } from 'react';

interface Command {
  cmd: string;
  exitCode: number;
  stdout: string;
  stderr: string;
}

interface TaskGroup {
  id: string;
  title: string;
  status: 'running' | 'success' | 'failed';
  summary?: string;
  commands: Command[];
}

interface AccordionProps {
  group: TaskGroup;
}

export function TaskGroupAccordion({ group }: AccordionProps) {
  // Automatically open the accordion if the task is currently running
  const [isOpen, setIsOpen] = useState<boolean>(group.status === 'running');

  // React to status changes (e.g., auto-expanding when a task starts)
  useEffect(() => {
    if (group.status === 'running') {
      setIsOpen(true);
    } else if (group.status === 'success') {
      setIsOpen(false); // Auto-collapse to clean up the chat feed on success
    }
  }, [group.status]);

  return (
    <div className={`task-card border rounded-md my-2 overflow-hidden ${group.status}`}>
      {/* Accordion Header */}
      <div 
        className="card-header flex items-center justify-between p-3 cursor-pointer bg-gray-50 select-none hover:bg-gray-100"
        onClick={() => setIsOpen(!isOpen)}
      >
        <div className="flex items-center space-x-3">
          <StatusIndicator status={group.status} />
          <span className="font-semibold text-sm text-gray-800">{group.title}</span>
        </div>
        <div className="flex items-center space-x-2 text-xs text-gray-500">
          <span>{group.commands.length} actions</span>
          <span className="transform transition-transform duration-200">
            {isOpen ? '▼' : '▶'}
          </span>
        </div>
      </div>
      
      {/* Accordion Content */}
      {isOpen && (
        <div className="card-content p-3 border-t border-gray-200 bg-white">
          {group.summary && (
            <p className="text-xs text-gray-600 mb-2 italic">{group.summary}</p>
          )}
          
          <div className="space-y-3">
            {group.commands.map((c, i) => (
              <div key={i} className="terminal-container rounded bg-gray-950 p-3 font-mono text-xs text-gray-200">
                <div className="flex items-center justify-between text-gray-500 border-b border-gray-800 pb-1 mb-2">
                  <span>$ {c.cmd}</span>
                  <span className={c.exitCode === 0 ? 'text-green-500' : 'text-red-500'}>
                    exit: {c.exitCode}
                  </span>
                </div>
                
                {/* Standard Outputs */}
                {c.stdout && (
                  <pre className="whitespace-pre-wrap text-emerald-400">{c.stdout}</pre>
                )}
                
                {/* Standard Errors */}
                {c.stderr && (
                  <pre className="whitespace-pre-wrap text-rose-400 mt-1">{c.stderr}</pre>
                )}
              </div>
            ))}
          </div>
        </div>
      )}
    </div>
  );
}

function StatusIndicator({ status }: { status: 'running' | 'success' | 'failed' }) {
  switch (status) {
    case 'running':
      return <div className="w-3 h-3 rounded-full bg-blue-500 animate-pulse" />;
    case 'success':
      return <span className="text-green-600 text-sm">✓</span>;
    case 'failed':
      return <span className="text-red-600 text-sm">✗</span>;
    default:
      return <div className="w-3 h-3 rounded-full bg-gray-300" />;
  }
}
